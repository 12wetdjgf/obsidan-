## 🐝 Stage 1: BeeCountingStage

### Bug #1: `NullPointerException`

**什么是NullPointerException？**

就是你对一个 `null`（空的、不存在的东西）调用了方法。比如：

```java
String s = null;
s.length();  // 💥 BOOM! NullPointerException!
```

**姐教你怎么想：**

Stack trace会告诉你错误发生在哪一行。但是！**错误发生的地方不一定是bug所在的地方！**

> 🔑 关键提示：去看看 **构造函数（constructor）**。有没有某个变量声明了但没有正确赋值？就像你买了个盒子但忘了把东西放进去一样～

比如说，你可能会看到类似这样的情况：

```java
// 这种写法有什么问题？
public MyClass(Scanner input) {
    Scanner input = new Scanner(System.in); // ← 🤔 嗯？？
    // 你创建了一个新的局部变量，而不是赋值给实例变量！
}
```

想想 **参数名** 和 **实例变量名** 的关系，只需改1行！

### Bug #2: `IndexOutOfBoundsException`

**什么是IndexOutOfBoundsException？**

就是你访问了数组/列表中不存在的位置：

```java
int[] arr = {1, 2, 3};  // 有效索引: 0, 1, 2
arr[3];  // 💥 BOOM! 只有3个元素，最大索引是2！
```

> 🔑 关键提示：**Java是0索引的！** 如果一个字符串长度是5，那有效索引是 `0, 1, 2, 3, 4`。检查循环边界条件，是不是用了 `<=` 而应该用 `<`？

只需改1行！

## 🔄 Stage 3: PalindromeStage（回文）

这个有 **两个bug**！先修简单的，再修难的。

### Bug #1: IntelliJ的黄色/橙色高亮

> IntelliJ超聪明的好吗！它会用黄色或橙色高亮提示你"这里有点可疑"。把鼠标悬停在 `digitsToIntList` 方法里那些高亮的地方，看看IDE跟你说了啥～

### Bug #2: 无限循环 😱

**怎么判断有没有无限循环？**

如果你运行debugger然后程序卡住了、不动了，那大概率就是死循环。

> 🔑 姐的调试技巧：
> 1. 在`while`循环里设个断点
> 2. 看循环条件是什么
> 3. 每次循环，条件变量有没有在朝着"退出循环"的方向变化？
> 4. 如果一个`while`循环永远满足条件，那就会一直转一直转...

```java
// 一个死循环的例子
int i = 10;
while (i > 0) {
    // 如果忘了 i-- 或者写成了 i++
    // 那 i 永远 > 0，永远出不去！
    doSomething();
    i++;  // ← 应该是 i-- 吧？？
}
```

看看循环退出条件，再看看每次迭代后变量是怎么变化的。

## 🤖 Stage 4: MachineStage

这个超级考验 **抽象思维** 的！

`sumOfElementwiseMax` 要做的事：
1. 两个数组，逐元素取最大值
2. 把这些最大值加起来

例子：
```
数组A: {2,  0, 10, 14}
数组B: {-5, 5, 20, 30}
逐元素max: {2, 5, 20, 30}
求和: 2 + 5 + 20 + 30 = 57
```

**重点来了：** 里面有两个神秘函数 `mysteryMax` 和 `mysteryAdd`，**不要进去看！不要试图理解它们！**

> 🔑 姐的策略：
> 1. 先手算一遍正确答案
> 2. 用debugger的 **Step Over**（不进入函数，只看结果）
> 3. 看 `arrayMax` 的返回值对不对 → 如果不对，说明 `mysteryMax` 有问题
> 4. 看 `arraySum` 的返回值对不对 → 如果不对，说明 `mysteryAdd` 有问题
> 5. 如果某个mystery函数坏了，**整个重写它**（用简单直白的方式）

```java
// 比如如果 mysteryMax 坏了，你可以直接写：
static int max(int a, int b) {
    return Math.max(a, b);
}
```

这就是 **抽象的力量** ——我不需要知道鱼的分子结构，也能判断这条鱼是不是死了🐟💀
